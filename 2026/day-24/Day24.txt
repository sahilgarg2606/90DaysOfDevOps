Switch back to main

git checkout main

Merge the branch

git merge feature-login

Result:

Git performs a **Fast-Forward Merge** because main had no new commits after the branch was created.

Example History

A --- B (main)

feature-login adds commits

A --- B --- C --- D (feature-login)

After merge

A --- B --- C --- D (main)

No merge commit is created.

# Notes Answers

## 1. What is a Fast-Forward Merge?

Fast-forward merge happens when the target branch has no new commits and Git can simply move the branch pointer forward.

Example

main: A --- B

feature: A --- B --- C --- D

After merge

main: A --- B --- C --- D

No merge commit is created.

---

## 2. When does Git create a Merge Commit?

Git creates a merge commit when both branches have different commits and their histories have diverged.

Example

main: A --- B --- C

feature: A --- B --- D

After merge

A --- B --- C -------- E
\              /
D ------------

E = merge commit

---

## 3. What is a Merge Conflict?

A merge conflict occurs when two branches modify the same line in the same file and Git cannot decide which change to keep.

Example conflict

<<<<<<< HEAD
Login button color = blue
=========================

Login button color = red

> > > > > > > feature-login

To resolve the conflict:

1. Edit the file and choose the correct code
2. Remove the conflict markers
3. Run

git add .

git commit -m "Resolve merge conflict"

1. What does squash merging do?

Squash merge:

Combines multiple commits from a branch

Creates one single commit in the target branch

Keeps history clean

Example

Feature branch

A --- B --- C --- D --- E

After squash merge

A --- F

F contains all changes from B, C, D, and E.

2. When would you use squash merge vs regular merge?

Use Squash Merge When:

Feature branch has many small commits

Commits are messy (typos, formatting fixes)

You want clean project history

Common in GitHub Pull Requests

Example

"Complete user profile feature"

instead of 10 small commits.

Use Regular Merge When:

You want to preserve commit history

Working in large team

Each commit has meaningful change

Debugging history later is important

Example

Tracking every step of development.

3. What is the trade-off of squashing?

Advantages

Cleaner history

Easier to read git log

Single commit per feature

Disadvantages

Individual commit history is lost

Harder to debug step-by-step changes

Cannot see detailed development process

Example

Instead of seeing

Fix bug
Update UI
Add validation

You only see

Add complete profile feature

What does cherry-pick do?

Cherry-pick allows you to apply a specific commit from one branch to another branch.

Instead of merging the whole branch, Git copies only the chosen commit.

Example

Feature branch

A --- B --- C --- D

Cherry-pick commit C

Main branch becomes

A --- C'

C' = copied commit

2. When would you use cherry-pick in a real project?

Cherry-pick is useful when you need only a specific fix or feature.

Common real-world scenarios

Production Hotfix

A bug fix is committed in a development branch, but production needs the fix immediately.

Instead of merging the entire branch, you cherry-pick only that commit.

Selective Feature Transfer

A commit from one feature branch is useful for another branch.

Backporting Fixes

A fix in a new version needs to be applied to an older release branch.

Example

release-v1.0
release-v2.0

You cherry-pick the bug fix from v2.0 to v1.0.

3. What can go wrong with cherry-picking?

Possible problems

Duplicate Commits

Cherry-pick creates new commit IDs, so the same change may appear multiple times.

Merge Conflicts

If the code in the target branch is different, cherry-pick may create conflicts that need manual resolution.

Harder History Tracking

Repeated cherry-picking can make the Git history confusing.

Important Git Commands

git checkout branch-name
git cherry-pick commit-hash
git log --oneline --graph
git cherry-pick --abort (cancel cherry-pick if conflict happens)
